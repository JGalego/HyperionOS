# Accessibility

## 1. Purpose

This document specifies accessibility as an architectural constraint on the
[13 — Dynamic UI Runtime](13-dynamic-ui-runtime.md), not a feature added to generated Workspaces
after the fact. Design Invariant 6 in [02 — Core Architecture §4](02-core-architecture.md#4-design-invariants)
states it plainly: "Accessibility is not a mode. Every Workspace generated by L6 must satisfy
[14 — Accessibility] constraints unconditionally." Concretely, this means every Panel the Workspace
Compiler produces (see [13 §5](13-dynamic-ui-runtime.md#5-algorithms)) carries a semantic
accessibility tree derived in the *same* compilation pass as its visual layout — not a
separately-authored, separately-maintained parallel UI — so that screen readers, switch devices,
eye-tracking systems, and voice control get first-class support automatically, for every Capability,
from day one, including Capabilities nobody has written yet.

## 2. Motivation

[01 — Vision & Philosophy §5](01-vision-and-philosophy.md#5-universal-usability-highest-priority)
places universal usability above raw capability whenever the two are in tension, and names people
with disabilities explicitly among the users Hyperion must feel natural to. The brief for this
system lists the concrete surface: screen readers, voice control, touch, keyboard, mouse, switch
devices, eye tracking, high contrast, color blindness, low vision, hearing impairments, dyslexia,
ADHD, motor impairments, and multilingual interaction. Two properties of Hyperion's architecture
make a bolt-on approach actively wrong here, rather than merely suboptimal:

1. **There is no fixed screen inventory to retrofit.** Because [13 — Dynamic UI Runtime](13-dynamic-ui-runtime.md)
   compiles a new Workspace graph for every Intent, an accessibility layer that only annotates
   pre-built screens would need to be rebuilt for every possible Intent/Capability combination —
   an unbounded set. Accessibility must instead be a property of the *compiler*, guaranteed for any
   graph it can produce.
2. **Voice control is not a convenience feature here — it is the primary interface.**
   [01 §7, Human Language First](01-vision-and-philosophy.md#7-human-language-first) already makes
   conversation the default way to operate Hyperion for every user. For a user who cannot use a
   mouse or touchscreen, that same conversational interface *is* their accessibility accommodation,
   which is why voice control is designed here as a first-class control surface feeding directly
   into [05 — Intent Engine](05-intent-engine.md), not as an add-on input method bolted onto a
   visual UI designed for pointing devices.

## 3. Architecture

Accessibility is produced as a byproduct of the same compiler pass described in
[13 §5.1](13-dynamic-ui-runtime.md#5-algorithms), then projected into modality-specific interfaces
and gated by a linter before anything reaches the cache:

```
        Capability UI Contract (declared per 24, 25 — includes accessibility_hints)
                                   │
                                   ▼
                    ┌───────────────────────────┐
                    │  Workspace Compiler (13)   │
                    └─────────────┬─────────────┘
                                   ▼
                    ┌────────────────────────────┐
                    │  Panel (visual layout)      │
                    │  + Accessibility Node        │  ← same compiler pass,
                    │    (parallel semantic tree)  │    isomorphic to Panel graph
                    └─────────────┬──────────────┘
                                   ▼
                 ┌──────────────────────────────────────┐
                 │   Accessibility Linter Gate            │  ← blocks non-conformant
                 │ (build-time CI + runtime canary sample)│    templates from cache
                 └─────────────────┬──────────────────────┘
                                   ▼
     ┌───────────┬─────────────────┼─────────────────┬───────────────┬─────────────┐
     ▼           ▼                 ▼                 ▼               ▼             ▼
  Visual      Screen-Reader     Switch-Scan        Eye-Gaze       Voice/Conv.    Real-Time
  Renderer    Projector         Renderer           Target Map     Control        Translation
  (13)                                                             (→ 05 Intent) Overlay (22)
     │           │                 │                 │               │             │
     └───────────┴────────┬────────┴─────────────────┴───────────────┴─────────────┘
                           ▼
              User Accessibility Profile
        (Semantic Object, encrypted at rest, 08 Memory / 16 Privacy)
```

Every modality-specific renderer projects from the *same* Accessibility Tree; none of them re-derive
semantics independently. This is what guarantees that a Capability author who declares one UI
contract gets screen-reader, switch, eye-gaze, and voice support simultaneously, without writing
five interfaces.

## 4. Data Structures

| Structure | Fields | Notes |
|---|---|---|
| `AccessibilityNode` | `node_id, panel_ref, role, accessible_name, description, state, actions[], relationships{}, language_tag, target_size, dwell_hint, live_region_priority` | One per Panel, plus finer sub-nodes for interactive sub-elements within a Panel. |
| `AccessibilityTree` | `tree_id, workspace_graph_id, nodes[], focus_order[]` | Isomorphic to the Workspace UI Graph in [13 §4](13-dynamic-ui-runtime.md#4-data-structures); never diverges structurally from it. |
| `UserAccessibilityProfile` | `profile_id, screen_reader_enabled, voice_control_enabled, dyslexia_font, pacing_multiplier, adhd_reduced_distraction_mode, motor_dwell_click_ms, target_size_multiplier, high_contrast_theme, color_blindness_mode, preferred_language, caption_style, alert_style (visual \| visual+haptic)` | A Semantic Object per [02 §2](02-core-architecture.md#2-shared-vocabulary), persisted via [08 — Memory Engine](08-memory-engine.md), access-controlled per [16 — Privacy Architecture](16-privacy-architecture.md). |
| `AccessibilityLintResult` | `template_id, passed, violations[]{rule_id, node_ref, severity}, timestamp` | Attached to the `CompiledLayoutTemplate` defined in [13 §4](13-dynamic-ui-runtime.md#4-data-structures). |
| `InputDeviceProfile` | `active_modalities[]` (switch, eye-tracker, voice mic, touch, keyboard, mouse) | Sourced from [20 — Device Framework](20-device-framework.md). |

## 5. Algorithms

**5.1 Tree derivation.** In the same compiler pass that produces a Panel's visual layout
([13 §5.1](13-dynamic-ui-runtime.md#5-algorithms)), the compiler reads the Capability's declared
`accessible_role`, `label_template`, and `keyboard_operations` (§6) together with the live-bound
Semantic Object's data (title, summary, state) and emits the corresponding `AccessibilityNode`.
Because tree and layout are produced together from the same inputs, they cannot structurally drift
— only their content, not their shape, can differ.

**5.2 Linter rule evaluation.** Before a `CompiledLayoutTemplate` is admitted to the cache in
[13 §5.4](13-dynamic-ui-runtime.md#5-algorithms), a fixed rule set is evaluated against every node:
accessible name present; contrast ratio ≥ 4.5:1 (or a profile-adjusted threshold for low vision);
target size ≥ 44 device-independent pixels × `profile.target_size_multiplier`; focus/scan order
forms a valid traversal with no unreachable node; any motion or auto-advance has a reduced-motion
alternative; a language tag is present on every text-bearing node. A template with any failing rule
is quarantined rather than cached (§9, §10).

**5.3 Modality projection.** For each modality present in the current `InputDeviceProfile`, a
projector reads the *same* `AccessibilityTree` and produces a modality-specific interface: the
screen-reader projector linearizes it via `focus_order`; the switch-scan projector groups nodes into
scan groups at a configurable speed; the eye-gaze projector sizes dwell targets by
`target_size_multiplier` and computes dwell timing from `dwell_hint`; the voice-control projector
auto-generates a grammar mapping each `accessible_name` to an action, which is submitted to
[05 — Intent Engine](05-intent-engine.md) as a micro-Intent rather than handled as a raw UI event —
"click submit" and "submit my answer" resolve identically.

**5.4 Real-time translation.** At render time, node text and bound Semantic Object content are
passed through a local translation Capability (see
[22 — Local AI Runtime](22-local-ai-runtime.md)) keyed to `profile.preferred_language`, cached by
`(content_hash, target_language)` to stay inside the latency budget in
[13 §11](13-dynamic-ui-runtime.md#11-performance-analysis). Cloud translation is used only with
explicit, per-instance consent per [16 — Privacy Architecture](16-privacy-architecture.md) — never
as a silent fallback, per Design Invariant 3 in
[02 §4](02-core-architecture.md#4-design-invariants).

**5.5 ADHD reduced-distraction mode.** Filters the Panel set to only Capabilities required for the
Intent's current step, hiding ancillary panels and suppressing non-essential live-update animation,
reversibly toggled by the user or profile default.

**5.6 Dyslexia typography and pacing.** Swaps rendered font family, letter/line spacing, and column
width tokens, and slows auto-advancing content (e.g., a flashcard timer from
[13's worked example](13-dynamic-ui-runtime.md#worked-example)) by `pacing_multiplier`, without
altering the underlying `AccessibilityTree` semantics — a screen reader user and a dyslexia-profile
user are reading the same tree through different projections.

**5.7 Hearing-impairment captioning and alerting.** Any Panel bound to a Capability that emits an
audio track (e.g. a video-playback or voice-call Capability) is compiled with a caption sidecar,
not an optional add-on: if the underlying Semantic Object already carries authored captions
(metadata per [09 — Knowledge Graph](09-knowledge-graph.md)), those are used directly; otherwise a
local speech-to-text model (per [22 — Local AI Runtime](22-local-ai-runtime.md)) generates timed
caption cues on first play, cached against the object so later playback doesn't re-transcribe.
Caption cues are re-synced to actual playback position on every seek or buffering stall — a
drifted caption is treated as a rendering bug, not an acceptable approximation, and is what
§13's "caption/audio sync" conformance check tests for. Separately, no Capability contract may
declare an audio-only alert (a completion chime, an error tone) without also declaring a visual
equivalent (a toast bound into the `AccessibilityTree` like any other node, per `caption_style`'s
sibling `alert_style` field) and, where the active device exposes a haptic actuator
([20 — Device Framework](20-device-framework.md)), a haptic pulse — the Accessibility Linter
(§13) rejects any registered alert Capability that declares only an audio channel, the same way it
rejects a Panel with no `accessible_role`.

## 6. Interfaces / APIs

```
AccessibilityTreeCompiler.derive(panel, capability_contract) -> AccessibilityNode
AccessibilityLinter.lint(template: CompiledLayoutTemplate)   -> AccessibilityLintResult
ModalityRenderer.project(tree, modality, profile)            -> ModalityInterface
VoiceControl.grammar(tree: AccessibilityTree)                -> IntentGrammar   # → 05
UserAccessibilityProfile.get() / .update(patch)               # 08-backed, 16-governed
```

The Capability UI Contract introduced in
[13 §6](13-dynamic-ui-runtime.md#6-interfaces--apis) is extended here with mandatory fields:
`accessible_role`, `label_template`, `keyboard_operations`, and `alt_text_hook`. Tooling in
[24 — Plugin Framework](24-plugin-framework.md) and [25 — SDK](25-sdk.md) rejects a Capability
publish that omits any of these — there is no code path by which a Capability becomes renderable
without also becoming accessible.

## 7. Pseudocode

```python
CONTRAST_MIN = 4.5
TARGET_SIZE_MIN_DP = 44

def lint_template(template):
    violations = []
    reachable = set()

    for node in template.tree.nodes:
        if not node.accessible_name:
            violations.append(Violation("accessible-name-present", node, "error"))

        ratio = contrast_ratio(node.foreground, node.background)
        if ratio < CONTRAST_MIN:
            violations.append(Violation("contrast-ratio", node, "error"))

        min_size = TARGET_SIZE_MIN_DP * node.profile_target_size_multiplier
        if node.is_interactive and node.target_size < min_size:
            violations.append(Violation("target-size", node, "error"))

        if node.has_motion and not node.reduced_motion_alternative:
            violations.append(Violation("motion-alternative", node, "warning"))

        if node.has_text and not node.language_tag:
            violations.append(Violation("language-tag-present", node, "error"))

    for node_id in template.tree.focus_order:
        reachable.add(node_id)
    unreachable = {n.node_id for n in template.tree.nodes if n.is_interactive} - reachable
    for node_id in unreachable:
        violations.append(Violation("focus-order-valid", node_id, "error"))

    passed = not any(v.severity == "error" for v in violations)
    if not passed:
        template = generate_fallback_template(template)   # §10: never fail closed
    return AccessibilityLintResult(template.template_id, passed, violations)
```

## 8. Security Considerations

`UserAccessibilityProfile` reveals disability- and health-adjacent information and is treated as
sensitive-category data: encrypted at rest, access-gated by capability token like any other
Semantic Object per [15 — Security Architecture](15-security-architecture.md), and never used for
inference or ranking outside the accessibility pipeline itself, per
[16 — Privacy Architecture](16-privacy-architecture.md). Switch-device and eye-tracker input crosses
a Trust Boundary from raw sensor/biometric data to Intent (see
[02 §2](02-core-architecture.md#2-shared-vocabulary)); that crossing is capability-gated and
audited exactly like any other Trust Boundary crossing, since gaze and calibration data are
themselves sensitive. Voice control's always-available micro-Intents require a liveness/focus check
so a command spoken off-screen or by another person cannot hijack a Workspace. Real-time translation
must not exfiltrate private Semantic Object content to a cloud translator without the explicit,
per-instance consent required by Design Invariant 3
([02 §4](02-core-architecture.md#4-design-invariants)) and
[16 — Privacy Architecture](16-privacy-architecture.md).

## 9. Failure Modes

- A Capability omits accessibility metadata from its UI contract.
- The `AccessibilityTree` and visual Panel graph drift during a rapid burst of live updates
  ([31 — Event System](31-event-system.md)), so a screen reader announces stale state.
- The linter has a heuristic gap and passes a template with a genuinely inaccessible pattern
  (a false negative).
- The local translation model is unavailable, slow, or mistranslates.
- Dwell-click misfires under motor tremor, causing an accidental activation.
- ADHD reduced-distraction mode hides a panel the user actually needed for the current step.

## 10. Recovery Mechanisms

Per Design Invariant 5, "degrade, never fail closed"
([02 §4](02-core-architecture.md#4-design-invariants)): when a Capability omits accessibility
metadata, the compiler deterministically derives a generic-but-valid role and name from the
Capability's own name rather than emitting a nameless node, so nothing is ever silently
inaccessible, only less precisely labeled. Drift between tree and visual state is corrected by
resyncing on the next event tick, since both are derived idempotently from the same underlying
Panel state. The linter runs twice: as a blocking CI/build gate before a template can be published,
and as a runtime canary sampling live templates post-ship — a canary failure automatically
quarantines the template and reverts to the last known-good version via
[33 — Rollback & Recovery](33-rollback-recovery.md). The user can always ask "why am I seeing this"
per [18 — Explainability & Trust](18-explainability-and-trust.md) and can always override any mode
or profile setting ("show me everything," disable reduced-distraction) per
[01 §6](01-vision-and-philosophy.md#6-adaptive-complexity). Dwell-click includes a
confirm-then-commit debounce sized from `motor_dwell_click_ms`. Translated content always carries a
"translated from X" disclosure with a one-tap revert to the original.

## 11. Performance Analysis

Because tree derivation happens inside the same compiler pass as
[13's layout synthesis](13-dynamic-ui-runtime.md#5-algorithms) rather than a second pass over the
finished UI, it adds negligible marginal latency to the sub-second generation budget in
[13 §11](13-dynamic-ui-runtime.md#11-performance-analysis) — tree nodes are a byproduct of template
compilation and are cached alongside the visual template, not recomputed per instantiation. Linting
is amortized at template-compile/publish time, so it never sits in the hot render path; only the
lightweight runtime canary sampling (§10) runs post-ship, on a small fraction of instantiations.
Translation is cached by content hash so repeated strings (menu labels, recurring Capability names)
are effectively free after first render. Modality projection (§5.3) is a single cheap linear pass
over an already-materialized tree — single-digit milliseconds even for a dense Workspace like the
eight-panel exam-prep example in
[13's Worked Example](13-dynamic-ui-runtime.md#worked-example).

## 12. Trade-offs

Deriving accessible names and roles heuristically from a Capability's declared contract trades some
hand-authored precision for a guarantee that *every* Capability, including ones no accessibility
specialist ever reviewed, produces a usable tree — the escape hatch is `label_template`, an explicit
author-supplied override that is still linted like anything else. A strictly blocking linter gate
trades some plugin-ecosystem velocity ([24 — Plugin Framework](24-plugin-framework.md)) for a
platform-wide guarantee; the cost is mitigated by running the same linter locally in
[25 — SDK](25-sdk.md) so failures surface at author-time rather than at review or, worse, at
runtime for an end user. Projecting every modality from one shared tree, rather than hand-building
five separate interfaces, keeps the test matrix tractable at the cost of each individual modality
being slightly less bespoke than a dedicated build would be. Reduced-distraction and other adaptive
modes trade some information completeness for focus, mitigated by the same transparency and
reversibility guarantees that govern Adaptive Complexity generally
([01 §6](01-vision-and-philosophy.md#6-adaptive-complexity),
[18 — Explainability & Trust](18-explainability-and-trust.md)). A [Compatibility Layer](27-compatibility-layer.md)
Workspace hosting opaque legacy-application pixels is the one named, bounded exception to
Invariant 6's "unconditionally": there is no Capability contract for this compiler pass to derive
an `AccessibilityTree` from. 27 addresses this directly rather than leaving it silent — a
platform-native accessibility bridge where the guest OS exposes one, degrading to pixel-level
magnification/contrast/OCR accommodations and a mandatory user-facing disclosure otherwise — and
this document's linter (§13) tracks that tier per Compatibility Workspace so the exception stays
visible and testable rather than becoming an unmonitored gap.

## 13. Testing Strategy

The Accessibility Linter is a mandatory gate in the shared CI harness defined in
[35 — Testing Strategy](35-testing-strategy.md), run on every `CompiledLayoutTemplate` before it
enters the cache in [13 §5.4](13-dynamic-ui-runtime.md#5-algorithms), plus runtime canary sampling
post-ship. A per-condition conformance matrix is maintained: screen-reader traversal correctness,
switch-scan path validity, contrast-ratio checks under each `high_contrast_theme` and
`color_blindness_mode`, target-size checks under each motor profile, dwell-click debounce timing,
translation round-trip fidelity, caption/audio sync for hearing-impairment accommodations, and —
for every active [Compatibility Layer](27-compatibility-layer.md) session — that
`accessibility_bridge` is `Platform` or `PixelFallback` with its disclosure chrome actually
rendered; a session recorded as `None` with no disclosure fails this gate exactly like a native
Workspace missing an `accessible_role`.
Fuzz testing generates random Intent/Capability combinations and asserts the "always produce a
valid fallback tree" guarantee in §5.1/§10 holds under all of them. Periodic structured sessions
with assistive-technology users feed refinements back into the heuristics in §5. Every
[32 — Update System](32-update-system.md) release is gated on this suite so a platform update can
never silently regress accessibility for an existing Workspace template.
